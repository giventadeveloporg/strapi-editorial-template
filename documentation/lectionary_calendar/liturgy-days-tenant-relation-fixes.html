Liturgy Days – Tenant Relation Fixes & Production Replication

This document describes the fixes applied so that Liturgical Calendar – Day entries are correctly linked to a tenant and visible to editors in Content Manager. It also provides guidelines to replicate the same setup in production.

1. Overview of the problem and fixes

After importing liturgy days with import-liturgy-days-from-pdf.js, the script reported “Created N liturgy days for tenant X”, but verify-editor-tenant-liturgy.js showed Liturgy days for this tenant: 0, and editors saw no entries in Content Manager. The root causes were:

  • Document Service (Strapi 5) may not persist the tenant manyToOne relation on documents(...).create() when the relation is passed as part of the payload.
  • Filter mismatch: The verify and delete scripts filtered by tenant: tenant.id only, while the Content Manager uses $or: [{ tenant: tenant.id }, { tenant: { documentId: tenant.documentId } }]. If the relation is stored by documentId in the document layer, filtering by numeric id alone returns no results.

Three fixes were applied so that created liturgy days are linked to the tenant and all scripts and the Content Manager see the same set of records.

2. Fix 1 – Import script (tenant relation + DB fallback)

Import script – tenant link and DB fallback

Script: import-liturgy-days-from-pdf.js

Changes:

  • Use tenant: tenant.id (numeric) in the create payload, matching the seed script and the way the entity table stores the relation.
  • After each successful documents(...).create(), call db.query('api::liturgy-day.liturgy-day').update({ where: { documentId: createdDoc.documentId }, data: { tenant: tenant.id } }) so the tenant relation is set on the entity table even if the Document Service did not persist it.

This ensures that every created liturgy day has its tenant_id (or equivalent) set in the database, so document API filters find it.

3. Fix 2 – Verify script (same filter as Content Manager)

Verify script – filter alignment

Script: verify-editor-tenant-liturgy.js

Change: When counting liturgy days per tenant, use the same filter as the content-manager extension:

filters: tenant.documentId != null
  ? { $or: [{ tenant: tenant.id }, { tenant: { documentId: tenant.documentId } }] }
  : { tenant: tenant.id }

So “Liturgy days for this tenant” matches exactly what the editor sees in the Content Manager list.

4. Fix 3 – Delete script (same filter)

Delete script – filter alignment

Script: delete-liturgy-days.js

Change: When resolving document IDs to delete for a tenant, use the same $or filter (by tenant.id and tenant.documentId) so the delete script removes the same set of documents that the verify script counts and the editor sees.

5. Tenant ↔ Liturgical Calendar relationship

How tenant and liturgy days are related
  • Content type: api::liturgy-day.liturgy-day (Liturgical Calendar – Day) has a manyToOne relation to api::tenant.tenant.
  • Storage: The relation is stored in the entity table (e.g. liturgy_days.tenant_id or via document engine link). Strapi 5 may store or query by numeric id or by documentId; the content-manager and scripts use $or so both are matched.
  • Editor visibility: The content-manager extension restricts the list view for editors (role strapi-editor) to the tenant assigned in Editor Tenant Assignment (api::editor-tenant.editor-tenant). Only liturgy days whose tenant matches that assignment are shown.
  • Assigning editors: Run assign-editor-to-directory-tenant.js with the editor’s email and the tenant ID so that editor sees that tenant’s liturgy days. After assignment, the editor must log out and log back in.

Scripts involved:

  • import-liturgy-days-from-pdf.js – creates liturgy days and sets tenant (including via DB fallback).
  • verify-editor-tenant-liturgy.js – shows editor → tenant and liturgy-day count per tenant (same filter as CM).
  • delete-liturgy-days.js – deletes liturgy days for a tenant or all (same filter).
  • assign-editor-to-directory-tenant.js – maps an editor email to a tenant so they see that tenant’s data in CM.
  • update-liturgy-days-malayalam-from-excel.js – updates only dayHeadingMalylm on existing liturgy days from an Excel file (date + dayHeadingMalylm columns).

6. Production replication guidelines

To replicate the same tenant–liturgy setup in production, follow these steps in order.

Step 1 – Ensure tenant exists

Create the production tenant in Strapi Admin (Content Manager → Tenant) if it does not exist. Note its tenantId (e.g. tenant_production_001).

Step 2 – (Optional) Delete existing liturgy days for that tenant

If you are re-importing, run the delete script so you do not duplicate entries. Stop the Strapi server before running.

Delete liturgy days for tenant
🔧 Customize parameters
node scripts/delete-liturgy-days.js --tenant-id=tenant_production_001
Windows single-line:
node scripts/delete-liturgy-days.js --tenant-id=tenant_production_001
Step 3 – Import liturgy days for the production tenant

Place the PDFs in documentation/lectionary_calendar/ (or set --en-pdf and --ml-pdf). Run the import with the production tenant ID and year. Strapi server must be stopped.

Import liturgy days
🔧 Customize parameters
node scripts/import-liturgy-days-from-pdf.js --tenant-id=tenant_production_001 --year=2026
Windows single-line:
node scripts/import-liturgy-days-from-pdf.js --tenant-id=tenant_production_001 --year=2026
Step 4 – Assign editors to the production tenant

For each editor who should see this tenant’s liturgy days, run the assign script with their login email and the same tenant ID. Use the two-argument form for Windows (no env vars):

Assign editor to tenant
🔧 Customize parameters
node scripts/assign-editor-to-directory-tenant.js editor@example.com tenant_production_001
Windows single-line:
node scripts/assign-editor-to-directory-tenant.js editor@example.com tenant_production_001
Step 5 – Verify tenant and liturgy counts

Run the verify script with an editor’s email to confirm their tenant and liturgy-day count. Strapi server must be stopped.

Verify editor tenant and liturgy
🔧 Customize parameters
node scripts/verify-editor-tenant-liturgy.js editor@example.com
Windows single-line:
node scripts/verify-editor-tenant-liturgy.js editor@example.com

Expect “Liturgy days for this tenant” to be greater than zero after a successful import. If the editor still sees 0 in Content Manager, have them log out and log back in.

✅ Production checklist
  • Tenant exists in Strapi (Content Manager → Tenant).
  • Strapi server stopped before running delete/import/verify.
  • Delete (optional) run for the target tenant before re-import.
  • Import run with --tenant-id=<production-tenant-id> and --year=YYYY.
  • Optionally run update-liturgy-days-malayalam-from-excel.js to fill dayHeadingMalylm from an Excel file (see section 7).
  • Each editor assigned via assign-editor-to-directory-tenant.js with email and tenant ID.
  • Verify script shows non-zero liturgy days for that tenant.
  • Editors log out and log back in after assignment.

7. Import liturgy days into production (Strapi Cloud)

To get your local Liturgical Calendar – Day data (Content Manager) into production Strapi Cloud, use one of the options below. Ensure STRAPI_CLOUD_URL and STRAPI_CLOUD_API_TOKEN (Full Access token) are set in .env (see .env.example).

Verify .env
  • STRAPI_CLOUD_URL – Production base URL (e.g. https://YOUR-PROJECT.strapiapp.com), no trailing slash.
  • STRAPI_CLOUD_API_TOKEN – Full Access API token from Cloud (Settings → API Tokens).
  • Same pattern is used by rest_api_push_to_cloud.js, fix_production_tenant_and_dates.js, and clear_cloud_data.js.
Option A – Full export + REST push (existing process)

Use when production is empty or you want to push all content (including liturgy days) from local to Cloud. Liturgy days are included in the export and pushed after tenants so the tenant relation can be linked.

  1. Stop local Strapi. Create export: npm run strapi export -- --no-encrypt -f my-export → my-export.tar.gz (if export reports 403 on assets, the archive is still valid).
  2. Push to Cloud: npm run push:rest-to-cloud -- my-export-4.tar.gz --uploads=local. Images are read from public/uploads and uploaded to Cloud — recommended when export could not bundle asset bytes. Env: STRAPI_CLOUD_URL, STRAPI_CLOUD_API_TOKEN. See Push to Cloud with images from local in the main transfer doc for the full command block and details.
  3. Liturgy days are pushed in dependency order (tenants first, then liturgy-days). Phase 2 restores the tenant relation on each liturgy day.

If Cloud already has data, consider Option C to push only liturgy days and avoid re-pushing everything.

Option B – Strapi transfer

Use when production is empty and you want to stream the full database and assets from local to Cloud (WebSocket-based).

  1. In Cloud Admin: Settings → Transfer Tokens → Create token (type Push or Full Access). Copy the token.
  2. On your machine (local Strapi running): npm run strapi transfer -- --to https://YOUR-PROJECT.strapiapp.com/admin --to-token YOUR_TOKEN --force.
  3. Confirm when prompted. Transfer replaces Cloud data and assets with local. Liturgy days are included.

If transfer fails (e.g. WebSocket closure), use Option A (REST push) instead; see documentation/strapi5_export_import_transfer.html.

Option C – Liturgy-only push (recommended when Cloud already has data)

Use when production already has (or should have) content and you want to add or sync liturgy days. The script:

  • Resolves tenant from the local entity table so tenantId is never undefined (fixes “tenant undefined not found on Cloud”).
  • Creates missing tenants on Cloud – if a tenant (e.g. tenant_demo_002) does not exist on Cloud, it is created from local data so no entries are skipped.
  • Upsert – for each liturgy day: if a record for the same date and tenant already exists on Cloud, it is updated (PUT); otherwise a new one is created (POST).
  • Follow-up PUT for tenant relation – after each create (POST), the script sends a PUT with tenant so the relation is set even if the Content API did not persist it on create. This ensures editors (who are filtered by tenant) see the entries.

Script: push-liturgy-days-to-cloud.js

  1. Stop local Strapi. Run: node scripts/push-liturgy-days-to-cloud.js (all local liturgy days) or node scripts/push-liturgy-days-to-cloud.js --tenant-id=tenant_demo_002 for one tenant.
  2. Use DRY_RUN=1 to log count and resolved tenantId without sending requests.
  3. Editor Tenant Assignment on Cloud: So that editors see liturgy days in Content Manager, create an Editor Tenant Assignment in Cloud Admin: Content Manager → Editor Tenant Assignment → Create new entry → set adminUserEmail to the editor’s login email (e.g. mosc.regular.user@keleno.com) and tenant to the same tenant (e.g. the one with tenantId tenant_demo_002). Save, then have the editor log out and log back in on the Cloud admin URL.
If editors still don’t see liturgy days (admin sees 166, editor sees 0): The tenant relation may not have been set on the created records. Delete liturgy days on Cloud and reimport:
  1. Delete on Cloud: node scripts/clear_cloud_data.js --remote --types=liturgy-days (uses STRAPI_CLOUD_URL and STRAPI_CLOUD_API_TOKEN). Optional: --dry-run to list count first.
  2. Re-run push: node scripts/push-liturgy-days-to-cloud.js --tenant-id=tenant_demo_002. The script now does a follow-up PUT after each create to set the tenant relation.
  3. On Cloud Admin, ensure Editor Tenant Assignment exists for the editor’s email and the same tenant, then have the editor log out and log back in.
Push liturgy days to Cloud
🔧 Customize parameters
node scripts/push-liturgy-days-to-cloud.js
Windows single-line:
node scripts/push-liturgy-days-to-cloud.js

8. Update Malayalam headings from Excel

After importing liturgy days from the English (and optionally Malayalam) PDFs, you can overwrite only the Malayalam day heading (dayHeadingMalylm) using an Excel file. The script does not create new records; it finds existing liturgy days by date for the given tenant and updates that single field. UTF-8 / Malayalam encoding is preserved when writing to Strapi.

What we did
  • Added script update-liturgy-days-malayalam-from-excel.js.
  • Input: Excel file with two columns – date and dayHeadingMalylm (e.g. documentation/lectionary_calendar/Liturgy-Days-Malayalam-2026.xlsx).
  • For each row, the script finds liturgy-day document(s) for that date and tenant, then updates only dayHeadingMalylm.
  • Dates in Excel can be Date type, Excel serial number, or string (e.g. YYYY-MM-DD or DD/MM/YYYY). Column names are matched case-insensitively.
Run the Malayalam update (Strapi stopped)

Required: --tenant-id=XXX. Optional: --excel=path (default: documentation/lectionary_calendar/Liturgy-Days-Malayalam-2026.xlsx), DRY_RUN=1 to preview without writing.

Update Malayalam headings from Excel
🔧 Customize parameters
node scripts/update-liturgy-days-malayalam-from-excel.js --tenant-id=tenant_production_001
Windows single-line:
node scripts/update-liturgy-days-malayalam-from-excel.js --tenant-id=tenant_production_001

9. Translation workflow: PDF to Malayalam (Google Nana Banana)

To produce the Malayalam Excel used in section 7, we used Google Nana Banana (or similar AI tool) to translate the English text from the PDF into Malayalam. The steps we followed:

Steps to get Malayalam text and generate the Excel
  1. Copy the text from the PDF file – Extract or copy the English liturgical day headings (and any other text you want translated) from the source PDF.
  2. Paste the text into Google Nana Banana with the instruction: Decode the text to malayalam language – The tool translated the content to Malayalam. You can also provide an image of the PDF page if the tool supports image input; we gave it the image and it translated everything for us.
  3. Generate the Excel sheet – Create a spreadsheet with two columns: date (YYYY-MM-DD or equivalent) and dayHeadingMalylm. Paste the translated Malayalam text into dayHeadingMalylm, one row per liturgy day. Save as documentation/lectionary_calendar/Liturgy-Days-Malayalam-2026.xlsx (or your chosen path) with UTF-8 / Unicode so Malayalam characters are preserved.

Using this workflow, we had everything translated from the PDF into Malayalam via the tool, then generated the Excel sheet and ran update-liturgy-days-malayalam-from-excel.js to update only the dayHeadingMalylm field in the Strapi database.
